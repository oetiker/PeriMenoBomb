# v1.1 — Report Views Design

**Date:** 2026-05-29
**Status:** Approved (brainstorming) — ready for implementation plan
**Supersedes:** the v1.1 scope in `docs/ROADMAP.md` and §7 of `docs/superpowers/specs/2026-05-27-perimenobomb-design.md`

## 1. Goal & Scope

Add a **report area** (`/report`) with three analyses on top of the existing entry data:

1. **Cycle heatmap** (`/report/cycle`) — pick an *anchor* event (e.g. period start) and a *value* symptom (e.g. headache); render a rotated, interactive heatmap where each column is one anchor occurrence over time and each row is a day-offset relative to the anchor, cell colour = the value symptom's intensity. Reveals patterns like "headache builds up 2–3 days before the period".
2. **"Days since last X"** (`/report/since`) — a counter of days since a chosen symptom's most recent occurrence, plus the list of intervals between occurrences with average / shortest / longest.
3. **Event list** (`/report/list`) — a day-grouped list of all recorded entries with every captured field, filterable by symptom and/or tag and date range, exportable to PDF.

**Important project context:** PeriMenoBomb has **no existing users**. No backward compatibility, redirects, or data retrofits are required. The default template and schema may change freely.

**Out of scope (→ v1.2):** CSV export · per-symptom 12-month calendar heatmap · bleeding-duration analysis (enabled by the new Start/End-Mens events but not built) · slider↔number colour-axis switcher · anchor minimum-spacing merge.

## 2. Foundation

Update all internal links, `goto()` calls, `BottomNav` hrefs, and any `route` value persisted by the `openDialog` store. No redirects (no users). The `BottomNav` "Verlauf" item is already wired to `/verlauf` — point it at `/report` instead. Guarded by existing tests + build.

### 2.2 Shared occurrence primitive

Both the heatmap (columns) and the since-view (intervals) consume **one** definition of "occurrence of symptom X":

```
listOccurrenceDates(symptomId): Promise<string[]>
  // distinct date keys (YYYY-MM-DD), ascending, where an entry row exists
  // for symptomId. Uses the existing `symptomId` index on db.entries.
```

This is robust because `upsertEntry` always writes a row (even with all-null/empty fields) and there is **no** "delete-if-empty" logic — saving an input-less event in the EntryEditor produces a persistent row. "Occurrence = entry row exists" therefore holds for pure events.

New date helper in `src/lib/utils/date.ts`:

```
daysBetweenKeys(a, b): number   // whole-day difference via fromDateKey, sign = a − b
```

Reuse the existing `listEntriesForRange(from, to)` for the event list and heatmap value lookups, and the existing `liveQuery` store for reactivity (views update live as new entries are saved).

### 2.3 Anchor events in the default template

Add two event symptoms to `src/lib/templates/perimeno-default.ts`:

- **Start Mens** — event, `comment` input enabled (optional), `daily: false`. Logged ad-hoc via the "+" picker.
- **Ende Mens** — same shape.

These give fresh installs the anchor the cycle heatmap depends on, and lay groundwork for future bleeding-duration analysis. Placement: a sensible group (e.g. a "Zyklus" root) — final naming/colour/icon decided in the plan.

## 3. Cycle Heatmap (`/report/cycle`)

### 3.1 Layout & axes

- **Columns** = each anchor occurrence, chronological, oldest left. (Per user decision: *every* occurrence = one column, unfiltered.)
- **Rows** = day-offset relative to the anchor. The **day-0 line** sits at the same height across all columns; negative offsets above (before the event → PMS window), positive offsets below.
- Two selectors at the top: **Anchor** (any symptom with ≥1 occurrence) and **Value symptom** (any symptom). Last choices persisted in `meta`.

### 3.2 Colour mapping (user choice: "everything, value when present")

Per cell, resolve the value symptom for that column's offset day:

- **Slider present** → saturation from `sliderValue` (domain **1–100**, fixed by `SliderInput`). `sliderValue === null` (the "unspez" slot) renders as a distinct **pale marker**, NOT "no entry".
- **else number present** → saturation from `numberValue` normalized over a **stable domain = the full selected date range**, never the visible viewport. (Viewport-relative normalization would change a value's colour on pan/zoom and destroy cross-cycle comparison.)
- **else (no value input)** → two-tone **recorded / not-recorded**.
- If a symptom has **both** slider and number, the **slider wins** by default (intensity). Number-as-colour is a v1.2 option.
- Cell states: no entry that day = neutral grey; anchor day = violet outline.

### 3.3 Anchor assumption & guard

The anchor is assumed **event-like** (period start). With "every occurrence = column", a symptom mistakenly logged on many consecutive days yields one column per day. Rather than silently rendering hundreds of columns, when occurrences exceed ~60 in the selected range the view shows a notice ("very many anchors — narrow the date range") and caps visibly. State the assumption explicitly in UI copy.

### 3.4 Overlap (by design)

Because each column is anchored to its own occurrence and cycle length varies, the same calendar day's value can appear in two adjacent columns (each at a different offset). This is intentional and documented — it's what lets irregular cycles align on day 0.

### 3.5 Interaction

- **Fixed axis gutters:** top = column date labels, left = day-offset labels. These stay pinned.
- **Only the plot cells** pan and zoom, via a CSS `transform` (translate + scale) on the plot layer.
- Axis tick labels are **repositioned from the current pan/zoom state** and culled when off-screen (spacing/size adapt to the zoom).
- This is the **hardest build** in v1.1. Implementation: isolate the transform/tick-projection logic into a small, unit-tested module (input: pan/zoom + data extents → cell rects + visible tick positions); the Svelte component is a thin renderer.
- **iOS risk:** pinch/pan in a standalone PWA is fragile (see the iOS-Safari memory). Mitigation: manual on-device test; if pinch proves flaky, fall back to zoom buttons + scroll.

## 4. Days Since Last X (`/report/since`)

The approved "Variant B".

- **Selector:** one symptom with ≥1 occurrence; last choice persisted in `meta`.
- **Counter:** `daysBetweenKeys(today, lastOccurrence)`, big number, with "letzte Erfassung: <date>" below. Uses `listOccurrenceDates`.
- **Interval list:** intervals between consecutive occurrences (from → to, N days), newest first. Below it: **Ø / shortest / longest**.
- **Edge cases:**
  - 0 occurrences → empty state "Noch keine Erfassung für dieses Symptom".
  - 1 occurrence → counter works; note "Noch kein Abstand — erst ab der zweiten Erfassung".
  - latest occurrence in the future → counter clamps to 0 with a subtle note.
- Read-only over the `symptomId` index; reactive via `liveQuery`.

## 5. Event List + PDF Export (`/report/list`)

### 5.1 On-screen view (mobile = cards)

Grouped by day: per-day section with a date heading; below it the day's entries as cards. Each card: symptom badge + name, slider value (with low/high reference), number value + unit, comment, tag chips. Newest first.

### 5.2 Filters

- **By symptom** (multi-select) and/or **by tag** (multi-select). Tag filter active → only entries for symptoms carrying at least one selected tag.
- **Date range** (from/to); default = full history.
- Filter state persisted in `meta`.

### 5.3 PDF export (tabular)

- Tabular, not cards: per-day section heading, then rows with columns **Symptom · Intensität · Anzahl · Kommentar · Tags**. Header: title, date range, generation date, active filters. Mirrors exactly the filtered view.
- **Library:** `jsPDF` + `jspdf-autotable` (handles table layout, pagination, header repetition). Deviates from the roadmap's `pdf-lib` mention — acceptable, no constraint. Versions verified at plan time.
- **Emojis:** not renderable in standard PDF fonts → PDF uses symptom name + a small colour square (symptom colour), no emoji. Umlauts/ß are covered by standard fonts.
- **iOS delivery:** `<a download>` is unreliable in an iOS standalone PWA. Delivery chain: **Web Share API** (`navigator.share` with file) when available → else open Blob in a new tab → else `<a download>`. Verify on-device (risk).

## 6. Testing Strategy

- **Unit (Vitest):** `listOccurrenceDates`, `daysBetweenKeys`; since-view interval + stats (avg/min/max); number normalization over the full range; filter logic (symptom/tag); column-building incl. dense-anchor cap; colour-mapping pure function.
- **Component (Vitest + @testing-library/svelte):** since-view (counter + list + 0/1-occurrence states); event list (day grouping + filters); heatmap cell colour.
- **E2E (Playwright, mobile viewport):** log Start/Ende-Mens + headache intensities → `/report/cycle` shows aligned columns; `/report/since` shows counter + intervals; `/report/list` filter by tag/symptom; PDF export produces bytes (function-level assertion).
- **Manual / on-device:** heatmap pinch/pan on iOS; PDF delivery on iOS.

## 7. Risks

1. **Fixed-axis pan/zoom** — hardest build; iOS pinch in standalone PWA. Mitigation: isolate + unit-test transform logic; on-device test; button+scroll fallback.
2. **PDF delivery on iOS** — Web-Share fallback chain.
3. **Number normalization** must use the full selected range, not the viewport.
4. **Dense anchor** — visible cap + notice.
5. **Variable cycle length** — same calendar day may appear in two adjacent columns (by design, documented).

## 8. Build Sequence
